📚 Concepts clés — Keycloak & OAuth2

🔑 Qu'est-ce que Keycloak ?

Identity Provider (IdP) Open Source qui centralise l'authentification :

  • 🔐 Gère les users et leurs credentials
  • 🎭 Distribue des JWT tokens après login
  • 🔗 Supporte OAuth2 & OpenID Connect (OIDC)
  • 🌍 API REST pour intégrations custom
🏛️ Qu'est-ce qu'un Realm ?

Espace isolé pour un projet/organisation :

  • master = realm admin (authentification interne)
  • logistock = realm app (nos utilisateurs métier)
  • Chaque realm a ses users, clients, roles, policies
  • Isolation complète : les users du realm A ne voient pas le realm B
🔄 Pourquoi 2 conteneurs ?
  • Keycloak (port 8080) = service auth
  • PostgreSQL (port 5433) = DB persistante

Keycloak a besoin d'une base de données pour stocker users, realms, sessions, etc. Sans PostgreSQL, les données seraient perdues au redémarrage du conteneur.

🔗 OAuth2/OIDC Flow

1. User ouvre http://localhost:5000 (web LogiStock)
2. App redirige vers http://localhost:8080/login
3. User tape admin/logistockadmin
4. Keycloak valide & retourne JWT token
5. User redirecté vers web avec token
6. Web peut accéder aux resources protégées

Étape 1 : Créer le realm, les utilisateurs et les mots de passe

💡 Script tout-en-un

Ce script fait les 4 opérations en séquence : token admin → realm logistock → 6 users avec mots de passe → attributs (email, prénom, nom). Il est idempotent : relancer ne crée pas de doublons.

powershell
# ── 1. Token admin ──────────────────────────────────────────────────────────
$tokenUrl = "http://localhost:8080/realms/master/protocol/openid-connect/token"
$body = @{
    client_id  = "admin-cli"
    username   = "admin"
    password   = "logistockadmin"
    grant_type = "password"
}
$response = Invoke-WebRequest -Uri $tokenUrl -Method Post -Body $body -UseBasicParsing
$TOKEN = ($response.Content | ConvertFrom-Json).access_token
Write-Host "✓ Token admin obtenu"

# ── 2. Créer le realm logistock ──────────────────────────────────────────────
$realmBody = @{
    realm       = "logistock"
    enabled     = $true
    displayName = "LogiStock IAM"
} | ConvertTo-Json

try {
    Invoke-WebRequest -Uri "http://localhost:8080/admin/realms" -Method Post `
        -Headers @{ Authorization = "Bearer $TOKEN"; "Content-Type" = "application/json" } `
        -Body $realmBody -UseBasicParsing | Out-Null
    Write-Host "✓ Realm 'logistock' créé"
} catch {
    if ($_.Exception.Response.StatusCode.value__ -eq 409) {
        Write-Host "⚠ Realm 'logistock' existe déjà"
    } else { throw }
}

# ── 3. Créer les 6 utilisateurs avec mots de passe ───────────────────────────
$users = @("alice", "bob", "charlie", "dave", "eve", "frank")
foreach ($username in $users) {
    $userBody = @{
        username    = $username
        enabled     = $true
        credentials = @(@{
            type      = "password"
            value     = "${username}123"
            temporary = $false
        })
    } | ConvertTo-Json -Depth 3

    try {
        Invoke-WebRequest -Uri "http://localhost:8080/admin/realms/logistock/users" -Method Post `
            -Headers @{ Authorization = "Bearer $TOKEN"; "Content-Type" = "application/json" } `
            -Body $userBody -UseBasicParsing | Out-Null
        Write-Host "✓ $username créé (mdp : ${username}123)"
    } catch {
        if ($_.Exception.Response.StatusCode.value__ -eq 409) {
            Write-Host "⚠ $username existe déjà"
        } else { throw }
    }
}

# ── 4. Mettre à jour email, prénom, nom ──────────────────────────────────────
$details = @(
    @{ username = "alice";   email = "alice@logistock.com";   firstName = "Alice";   lastName = "Martin" }
    @{ username = "bob";     email = "bob@logistock.com";     firstName = "Bob";     lastName = "Dupont" }
    @{ username = "charlie"; email = "charlie@logistock.com"; firstName = "Charlie"; lastName = "Lambert" }
    @{ username = "dave";    email = "dave@logistock.com";    firstName = "Dave";    lastName = "Bernard" }
    @{ username = "eve";     email = "eve@logistock.com";     firstName = "Eve";     lastName = "Dubois" }
    @{ username = "frank";   email = "frank@logistock.com";   firstName = "Frank";   lastName = "Moreau" }
)

foreach ($u in $details) {
    $userId = ((Invoke-WebRequest -Uri "http://localhost:8080/admin/realms/logistock/users?username=$($u.username)" `
        -Headers @{ Authorization = "Bearer $TOKEN" } -UseBasicParsing).Content | ConvertFrom-Json)[0].id
    $updateBody = @{
        firstName     = $u.firstName
        lastName      = $u.lastName
        email         = $u.email
        emailVerified = $true
    } | ConvertTo-Json
    Invoke-WebRequest -Uri "http://localhost:8080/admin/realms/logistock/users/$userId" -Method Put `
        -Headers @{ Authorization = "Bearer $TOKEN"; "Content-Type" = "application/json" } `
        -Body $updateBody -UseBasicParsing | Out-Null
    Write-Host "✅ $($u.username) — attributs mis à jour"
}

Write-Host "`n🎉 Realm + users prêts. Connectez-vous sur http://localhost:8080/realms/logistock/account avec alice/alice123"

Étape 3 : Créer les rôles (avec vérification robuste)

⚠️ Éviter l’erreur 400 Bad Request

L’API Keycloak retourne une erreur 400 si le rôle existe déjà ou si le corps JSON contient des champs superflus. Le script suivant rafraîchit le token, n’envoie que {"name": "..."} et vérifie l’existence préalable.

powershell
# Rafraîchir le token (au cas où)
$tokenUrl = "http://localhost:8080/realms/master/protocol/openid-connect/token"
$tokenBody = @{
    client_id = "admin-cli"
    username = "admin"
    password = "logistockadmin"
    grant_type = "password"
}
$response = Invoke-WebRequest -Uri $tokenUrl -Method Post -Body $tokenBody -ContentType "application/x-www-form-urlencoded"
$TOKEN = ($response.Content | ConvertFrom-Json).access_token

# Récupérer les rôles existants
$existingRoles = Invoke-WebRequest -Uri "http://localhost:8080/admin/realms/logistock/roles" `
    -Headers @{ "Authorization" = "Bearer $TOKEN" } | ConvertFrom-Json
$existingRoleNames = $existingRoles.name

# Liste des rôles (sans description)
$roles = @("admin", "stock_manager", "developer", "accountant", "viewer")

foreach ($roleName in $roles) {
    if ($roleName -in $existingRoleNames) {
        Write-Host "⚠️ Role '$roleName' existe déjà, ignoré" -ForegroundColor Yellow
        continue
    }
    $roleBody = @{ name = $roleName } | ConvertTo-Json
    try {
        Invoke-WebRequest -Uri "http://localhost:8080/admin/realms/logistock/roles" -Method Post `
            -Headers @{ "Authorization" = "Bearer $TOKEN"; "Content-Type" = "application/json" } `
            -Body $roleBody -ErrorAction Stop | Out-Null
        Write-Host "✅ Role créé: $roleName" -ForegroundColor Green
    } catch {
        $statusCode = $_.Exception.Response.StatusCode.value__
        $responseBody = $_.ErrorDetails.Message
        Write-Host "❌ Échec création role '$roleName' (HTTP $statusCode) : $responseBody" -ForegroundColor Red
    }
}

        clientId                  = "logistock-proxy"
        publicClient              = $false

            Write-Host "⚠ Client '$($client.clientId)' existe déjà"

⚠️ Données manquantes – Attribuer les groupes et rôles

Les utilisateurs, groupes et rôles ont été créés, mais il manque les associations :

  • ❌ Utilisateurs → groupes (ex. alice dans admins)
  • ❌ Groupes → rôles (ex. le groupe admins possède le rôle admin)

Exécutez les commandes PowerShell ci-dessous pour compléter la configuration.

powershell
# 1. Récupérer le token admin (s’il n’est pas déjà en mémoire)
$tokenUrl = "http://localhost:8080/realms/master/protocol/openid-connect/token"
$tokenBody = @{
    client_id = "admin-cli"
    username = "admin"
    password = "logistockadmin"
    grant_type = "password"
}
$response = Invoke-WebRequest -Uri $tokenUrl -Method Post -Body $tokenBody -ContentType "application/x-www-form-urlencoded"
$TOKEN = ($response.Content | ConvertFrom-Json).access_token
Write-Host "✓ Token admin récupéré"

# 2. Récupérer les ID des groupes
$groupsUrl = "http://localhost:8080/admin/realms/logistock/groups"
$groups = Invoke-WebRequest -Uri $groupsUrl -Headers @{ Authorization = "Bearer $TOKEN" } | ConvertFrom-Json
$groupIds = @{}
foreach ($g in $groups) { $groupIds[$g.name] = $g.id }

# 3. Récupérer les ID des utilisateurs
$usersUrl = "http://localhost:8080/admin/realms/logistock/users"
$users = Invoke-WebRequest -Uri $usersUrl -Headers @{ Authorization = "Bearer $TOKEN" } | ConvertFrom-Json
$userIds = @{}
foreach ($u in $users) { $userIds[$u.username] = $u.id }

# 4. Attribuer chaque utilisateur à son groupe
$mapping = @{
    "alice"   = "admins"
    "bob"     = "admins"
    "charlie" = "managers"
    "dave"    = "developers"
    "eve"     = "accountants"
    "frank"   = "viewers"
}

foreach ($user in $mapping.Keys) {
    $groupId = $groupIds[$mapping[$user]]
    $userId = $userIds[$user]
    $addUrl = "http://localhost:8080/admin/realms/logistock/users/$userId/groups/$groupId"
    Invoke-WebRequest -Uri $addUrl -Method Put -Headers @{ Authorization = "Bearer $TOKEN" } | Out-Null
    Write-Host "✓ $user → groupe $($mapping[$user])"
}

# 5. Attribuer les rôles aux groupes
# Récupérer les ID des rôles
$rolesUrl = "http://localhost:8080/admin/realms/logistock/roles"
$roles = Invoke-WebRequest -Uri $rolesUrl -Headers @{ Authorization = "Bearer $TOKEN" } | ConvertFrom-Json
$roleIds = @{}
foreach ($r in $roles) { $roleIds[$r.name] = $r.id }

# Mappage groupe → rôle
$groupRoles = @{
    "admins"      = "admin"
    "managers"    = "stock_manager"
    "developers"  = "developer"
    "accountants" = "accountant"
    "viewers"     = "viewer"
}

foreach ($groupName in $groupRoles.Keys) {
    $groupId = $groupIds[$groupName]
    $roleName = $groupRoles[$groupName]
    $roleId = $roleIds[$roleName]
    
    # URL pour ajouter un rôle à un groupe
    $addRoleUrl = "http://localhost:8080/admin/realms/logistock/groups/$groupId/role-mappings/realm"
    $body = @(@{ id = $roleId; name = $roleName }) | ConvertTo-Json
    Invoke-WebRequest -Uri $addRoleUrl -Method Post -Headers @{ Authorization = "Bearer $TOKEN"; "Content-Type" = "application/json" } -Body $body | Out-Null
    Write-Host "✓ Groupe $groupName → rôle $roleName"
}

Write-Host "`n✅ Tous les utilisateurs sont maintenant dans leurs groupes et les groupes ont leurs rôles."

Étape 6 : Tester le Refresh Token (PowerShell)

📚 Access Token vs Refresh Token

Access Token : durée courte (5-15 min) · contient les claims · stocké en mémoire (jamais localStorage).

Refresh Token : durée longue (1h-30j) · sert à renouveler l'access token · stocké en HttpOnly cookie · révocable.

powershell
# ─── Tester le Refresh Token ───────────────────────────────────────────

# 1. Obtenir access_token + refresh_token pour Alice
$body = @{
    client_id     = "logistock-api"
    client_secret = "api-secret-change-me"
    grant_type    = "password"
    username      = "alice"
    password      = "alice123"
}

🚨 LocalStack Community — Limitation IAM importante

La version gratuite de LocalStack (localstack:3.0.0 dans notre docker-compose) n'applique pas les politiques IAM.

  • Les opérations CRUD (créer users, groupes, politiques) fonctionnent
  • L'enforcement des politiques (Access Denied) nécessite LocalStack Pro via ENFORCE_IAM=1

👉 Dans ce TP on observe d'abord le mode permissif, puis on active ENFORCE_IAM=1 pour simuler le comportement strict. Les commandes CRUD fonctionnent dans les deux modes.

🔗 Du TP1 au TP2 – Authentification ≠ Autorisation

TP1 (Keycloak) : Alice s'authentifie → obtient un JWT (prouve son identité).
Mais ce JWT ne dit pas si elle peut lire un fichier S3 ou lancer une instance EC2.

TP2 (AWS IAM) : C'est le moteur d'autorisation qui répond « Alice a-t-elle le droit de faire cette action sur cette ressource ? »
Keycloak et IAM ne se remplacent pas : ils sont complémentaires dans toute architecture IAM moderne.

🎯 Objectif du TP2 : attacher des politiques IAM aux utilisateurs LogiStock (via groupes) pour contrôler leur accès aux ressources AWS simulées par LocalStack.

🎯 Objectif – Autorisation avec AWS IAM

Dans le TP1, vous avez centralisé l’authentification avec Keycloak (SSO, MFA, JWT).
Dans ce TP2, vous allez gérer les permissions fines sur les ressources métier (fichiers S3, serveurs EC2).

Concrètement :

  • Alice (CEO) pourra lire et écrire dans le bucket S3 logistock-data.
  • Frank (stagiaire) pourra seulement lire (pas modifier).
  • Chaque permission est définie par une politique IAM (fichier JSON) attachée à son groupe.

⚠️ Important : AWS IAM ne gère pas l’authentification des humains (c’est Keycloak). IAM gère l’autorisation une fois que l’identité est connue (via le JWT).

📚 Concepts clés — AWS IAM & LocalStack

🏢 Qu'est-ce que AWS IAM ?

Identity and Access Management — contrôle qui peut faire quoi sur les ressources AWS :

  • Users : identités individuelles
  • Groups : collections de users
  • Roles : permissions réutilisables
  • Policies : règles JSON qui autorisent/interdisent actions
🎯 Qu'est-ce que LocalStack ?

AWS mock local — simule les services AWS sur votre machine :

  • ✅ API 100% compatible avec AWS
  • ✅ Pas besoin de compte AWS
  • ✅ Gratuit et reproductible
🔐 Hiérarchie IAM (Least Privilege)

1. User alice
2. ↓ in Group "managers"
3. ↓ has Role "stock-manager"
4. ↓ with Policy = allow S3:GetObject on bucket/*
5. → alice peut lire (mais pas modifier) les fichiers S3

Principe : Donner juste les permissions nécessaires, rien de plus.

Keycloak (TP1) dit qui est Alice. AWS IAM (TP2) dit ce qu’elle peut faire sur les ressources.

Partie 2 : Implémenter PKCE (45 min)

📚 Qu'est-ce que PKCE ?

PKCE (RFC 7636) : Protection contre les attaques "Authorization Code Interception".

Problème sans PKCE : Un attaquant qui intercepte l'authorization code peut l'échanger contre un token.

Solution PKCE : Le client génère un code_verifier aléatoire, calcule son hash SHA256 (code_challenge), et l'envoie avec la requête. Keycloak vérifie que le code_verifier final correspond.

⚠️ Pré-requis — activer PKCE sur logistock-web

Console Keycloak → Clients → logistock-web → Settings → Proof Key for Code Exchange → S256.

powershell
# ─── PKCE — 4 étapes en PowerShell ─────────────────────────────────────

# ── Étape 1 : Générer code_verifier (32 octets, base64url) ─────────────
# Compatibilité PS 5.x et PS 7.x : instancier RNGCryptoServiceProvider
$rng   = [System.Security.Cryptography.RNGCryptoServiceProvider]::new()
$bytes = New-Object byte[] 32
$rng.GetBytes($bytes)
$CODE_VERIFIER = [Convert]::ToBase64String($bytes).TrimEnd("=").Replace("+","-").Replace("/","_")

# ── Étape 2 : Calculer code_challenge (SHA-256, base64url) ─────────────
$sha256         = [System.Security.Cryptography.SHA256]::Create()
$hash           = $sha256.ComputeHash([System.Text.Encoding]::ASCII.GetBytes($CODE_VERIFIER))
$CODE_CHALLENGE = [Convert]::ToBase64String($hash).TrimEnd("=").Replace("+","-").Replace("/","_")

Write-Host "code_verifier  : $CODE_VERIFIER"
Write-Host "code_challenge : $CODE_CHALLENGE"
Write-Host ""
Write-Host "⚠️  Ne confondez pas ces deux valeurs :" -ForegroundColor Yellow
Write-Host "   code_verifier  → gardé SECRET dans ce script" -ForegroundColor Yellow
Write-Host "   code_challenge → envoyé à Keycloak dans l'URL" -ForegroundColor Yellow

Partie 3 : Valider les tokens JWT (30 min)

⚠️ CRITICAL : Toujours valider les JWT

Danger : Un JWT peut être modifié côté client. L'API backend DOIT vérifier :

  • ✅ Signature (avec clé publique Keycloak)
  • ✅ Expiration (exp)
  • ✅ Issuer (iss = http://localhost:8080/realms/logistock)
  • ✅ Audience (aud)
🔍 Pourquoi ces 4 vérifications ?
  • Signature → garantit que le token n’a pas été modifié (intégrité) et qu’il provient bien de Keycloak.
  • Expiration (exp) → évite de réutiliser un token volé après sa durée de vie (5-15 min).
  • Issuer (iss) → évite qu’un attaquant utilise un token émis par un autre IdP (ex: faux Keycloak).
  • Audience (aud) → évite qu’un token destiné à un autre service (ex: logistock-web) soit accepté par votre API.

📚 Concepts clés — RBAC vs ABAC avec OPA

🎭 RBAC (Role-Based Access Control)

Permissions basées sur le RÔLE :

  • IF user.role == "stock_manager" → allow update_stock
  • Simple et prévisible
  • Mais inflexible (pas de contexte)

Problème : manager ne peut éditer stock que entre 9h-17h pendant jours ouvrables ?

Permissions basées sur les ATTRIBUTS :

  • IF user.role == "manager" AND user.department == "stock" AND current_time between 9h-17h AND user.location == "office" → allow
  • Très flexible et précis
  • Complexe à maintenir

Policy Engine — moteur décisionnel universel :

  • 📜 Policies = fichiers .rego
  • 🔄 Évalue requêtes → accept/deny
  • 🎯 Agnostique : fonctionne partout (Kubernetes, API, etc.)

Avantage : Granularité maximale, contexte

Limite : Plus complexe à gérer

📚 Concepts clés — JIT Access & Vault

⏰ JIT Access (Just-In-Time)

Accès temporaire sur demande — au lieu de credentials permanents :

  • ❌ Ancien : alice a une clé SSH valide 1 an
  • ✅ Nouveau : alice demande accès → Vault génère OTP 30 min
  • ⏱️ Après 30 min : OTP expirée, accès fermé
  • 📝 Audit : qui s'est connecté quand exactement
🔐 Permanent vs Dynamic Credentials

Permanent (risky) :

  • Valid 365 jours/an
  • Si volée → danger permanent
  • Rotation manuelle (pénible)

Dynamic (secure) :

  • Généré à la demande
  • Expire automatiquement (30 min)
  • Audit fin des accès
  • Rotation = creation automatique
🤖 Qu'est-ce que Vault ?

Secrets Management Engine :

  • 🔑 Stocke & gère secrets (passwords, API keys, SSH keys)
  • 🎫 Émet credentials dynamiques (OTP SSH, DB passwords)
  • 📜 Policies pour contrôler qui peut demander quoi
  • 📝 Audit trail complet

🎯 Objectif

Concepts couverts : Zero Trust principles, Device Posture, Context-Aware Access, Identity-Aware Proxy, Continuous Authorization, Compliance Score.

Partie 1 : Comprendre Zero Trust (30 min)

📚 Zero Trust Principles

"Never trust, always verify"

Ancien modèle (Castle-and-Moat) :

  • Réseau interne = confiance totale
  • Réseau externe = danger
  • Firewall = frontière

Problème : Si un attaquant entre (phishing, VPN compromise), il a accès à tout.

Zero Trust :

  • ❌ Pas de confiance par défaut (même réseau interne)
  • ✅ Vérifier CHAQUE requête (identity, device, context)
  • ✅ Least Privilege (accès minimal nécessaire)
  • ✅ Micro-segmentation (segments granulaires)
